三角洲摸红微信小游戏

一、项目概述
1. 项目类型：轻度模拟搜刮 + 随机反馈
2. 平台：微信小游戏
3. 核心卖点：短时爽感、结果可视化、价值累积
4. 目标用户：偏好开箱/抽卡/搜刮体验的轻度玩家

二、项目目标
1. 建立“选择容器—等待—开奖—结算”的核心闭环
2. 形成可扩展的物资体系与数值框架
3. 保持轻量开发成本，适配微信小游戏环境

三、核心玩法与循环
1. 选择容器：保险柜、电脑等
2. 搜刮阶段：触发转圈等待
3. 掉落展示：展示物资与价值
4. 结算累计：总价值增长，继续搜刮或退出

四、物资与容器设计
1. 容器类型
   - 保险柜：高价值概率，等待时间长
   - 电脑：中等价值概率，等待时间中等
2. 物资分级
   - 普通 / 中等 / 稀有 / 传说
3. 展示要素
   - 物资图片、单件价值、累计总价值

五、保险柜转圈时间规则（示例）
1. 普通物资：2–4秒
2. 中等物资：5–7秒
3. 稀有物资：8–12秒
4. 传说物资：13–18秒
5. 规则说明：等待时间越长，物资价值越高，时间区间可加入轻微随机浮动

六、经济与数值框架（示例）
1. 价值区间
   - 普通：1K–20K
   - 中等：20K–300K
   - 稀有：300K–3M
   - 传说：3M–15M
   价值区间要根据三角洲的物品价值来更新(暂时先写价格 先不随着三角洲金币系统更新---接口无法破解->因为还不会 哈哈)
2. 掉落权重：普通 > 中等 > 稀有 > 传说
3. 目标体验：高频小奖励 + 低频大惊喜

七、UI与交互
1. 主界面结构：容器列表 + 物资展示区 + 总价值
2. 动效反馈：转圈动画、价值跳字、掉落音效
3. 操作路径：2步内进入搜刮

八、技术方案
1. 方案对比
   - Unity 导出微信小游戏：适合 3D 或重度项目，上手快但包体与性能成本高
   - Cocos Creator：2D 友好，生态成熟，性能与包体更可控
   - 原生小游戏：成本最低、体积最小，适合轻度玩法快速迭代
2. 首版建议：直接做网络版本，采用原生或 Cocos，兼顾体积与性能
3. 架构建议：前端负责表现与操作，服务器负责匹配、掉落与结算
4. 数据配置：物资表、权重、价值区间由服务端配置与下发
5. 资源管理：物资图集 + 容器图 + 转圈动效

十、开发排期（建议）
1. 第1周：联机基础（账号与会话、匹配/房间、掉落接口）
2. 第2周：核心玩法联机化（转圈同步、掉落展示、结算）
3. 第3周：数值平衡 + 体验优化 + 上线准备

十一、联机版本首版设计（直接上线）
1. 核心流程：
   - 进入主界面 → 选择容器 → 匹配或好友组队 → 转圈等待 → 服务器掉落 → 结算与排行
2. 必做功能点：
   - 匹配/房间：随机匹配与好友组队
   - 服务端掉落：由服务器返回掉落结果与价值
   - 结算与排行：本局排行 + 总价值累计
   - 数据同步：转圈进度与开奖结果一致
3. 关键接口（示例）：
   - /match/start
   - /room/create
   - /room/join
   - /loot/roll
   - /match/finish
4. 同步与公平：
   - 掉落与结算由服务端计算
   - 客户端仅展示结果与动画

十二、功能完善清单（分级）
1. 高优先级
   - 每日/每周任务：完成任务获得额外搜刮次数或奖励
   - 收集册：物资图鉴与稀有物品解锁展示
   - 排行榜：当日/本周/历史最高价值
   - 保底机制：一定次数后必出稀有或更高
2. 中优先级
   - 账号等级：总价值累计解锁新容器
   - 容器升级：等待时间缩短或掉率提升
   - 成就系统：连续稀有、单局高价值等奖励
3. 社交与联机
   - 好友对战：同场对比带出价值
   - 观战/回放：查看他人开出内容
   - 小队奖励：组队完成目标获得额外掉落
4. 地区排名与排名
   - 地区榜：按省/市维度展示排行
   - 赛季榜：周期结算与奖励
   - 本地榜：附近或同城排名展示
